\begin{center}
  {\large\bfseries \textcolor{nfsuRed}{ABSTRACT}}
\end{center}

\vspace{0.5cm}

The Domain Name System (DNS) forms a critical backbone of the modern internet, yet it remains a frequent target for abuse, misconfiguration, and malicious exploitation. This dissertation presents \textbf{DNS Guard}, a hybrid domain threat intelligence platform that combines machine learning-based threat detection with comprehensive DNS configuration analysis and actionable remediation guidance.

The system performs multi-dimensional analysis of any given domain, extracting a 40-feature vector encompassing lexical characteristics, DNS record structure, DNSSEC validation, and composite security indicators. An ensemble model combining a Random Forest classifier with a Multi-Layer Perceptron (MLP) neural network produces a 0--100 threat score, accompanied by SHAP-style feature importance explanations for interpretability.

Beyond scoring, DNS Guard integrates a suite of security intelligence modules including WHOIS analysis with change detection, Certificate Transparency log monitoring, passive DNS history with fast-flux detection, typosquatting detection against top global brands, subdomain enumeration, reverse IP lookup, and real-time blocklist checks via Spamhaus DBL and PhishTank. IP geolocation is visualised on an interactive SVG world map, and comprehensive PDF reports are generated for each scan.

The platform further incorporates a context-aware AI chatbot powered by a Groq API (LLaMA 3.1 8B Instant), with a rule-based fallback engine covering key DNS security topics. A Retrieval - Augmented Generation (RAG) pipeline backed by a Qdrant vector database enables the chatbot to provide accurate, domain-specific security guidance.

The system is deployed as a containerised microservices architecture using Docker Compose, comprising a Next.js 14 frontend, a FastAPI backend with Celery task queuing, a dedicated AI/ML service, PostgreSQL for persistence, and Redis as the message broker. Evaluation demonstrates that DNS Guard effectively identifies malicious, misconfigured, and suspicious domains with high accuracy, offering a practical and extensible tool for DNS security monitoring and threat intelligence.

\vspace{0.5cm}
\noindent\textbf{Keywords:} DNS Security, Threat Intelligence, Machine Learning, DNSSEC, Typosquatting, Fast-Flux Detection, Phishing Detection, Natural Language Processing, RAG, Cybersecurity.

\section{Background}

The internet today connects billions of people, devices, and services across the world. At the heart of this connectivity is a system called the Domain Name System (DNS). DNS works like a phone book for the internet. When a user types a website address such as \texttt{www.example.com} into a browser, DNS translates that human-readable name into a numerical IP address that computers use to find and connect to the right server \cite{mockapetris1987rfc1034}. Without DNS, users would have to remember long strings of numbers to visit any website, which would make the internet very difficult to use.

Because DNS is so important, it has become a major target for attackers. Over the years, criminals have found many ways to misuse DNS to carry out attacks. Some attackers register domain names that look very similar to well-known brands in order to trick users into visiting fake websites. This is called typosquatting \cite{banerjee2008typosquatting}. Others use DNS to spread malware, run phishing campaigns, or hide the location of their command-and-control servers. Fast-flux DNS is one such technique where attackers rapidly change the IP addresses linked to a domain to avoid being blocked \cite{holz2008fastflux}.

At the same time, many organisations fail to properly configure their DNS records. Missing or incorrect SPF, DKIM, and DMARC records leave email systems open to spoofing attacks \cite{kitterman2011spf}. Domains without DNSSEC are vulnerable to cache poisoning, where an attacker can redirect users to a fake website without them knowing \cite{arends2005rfc4033}. These misconfigurations are very common and often go unnoticed until an attack has already happened.

Security researchers and network administrators need tools that can quickly check a domain for all of these issues in one place. Currently, most available tools only check one or two things at a time. There is a clear need for a platform that brings together DNS analysis, threat intelligence, machine learning-based scoring, and clear guidance on how to fix problems.

\section{Problem Statement}

Despite the critical role that DNS plays in internet infrastructure, there is a lack of accessible and comprehensive tools that can analyse a domain from multiple security angles at once. Existing solutions tend to be either too narrow in scope, too expensive for individual researchers and small organisations, or too technical for non-expert users to understand.

The specific problems this project addresses are as follows:

\begin{itemize}
    \item DNS abuse, including phishing, typosquatting, and fast-flux attacks, continues to grow every year \cite{apwg2023report}.
    \item Many domain owners do not know whether their DNS configuration is correct or secure.
    \item There is no single open tool that combines DNS record analysis, WHOIS data, certificate transparency, passive DNS history, threat intelligence feeds, and machine learning-based risk scoring in one platform.
    \item When a domain is flagged as suspicious, users often do not receive clear guidance on what the problem is or how to fix it.
\end{itemize}

This project aims to solve these problems by building DNS Guard, a web-based platform that performs a full security analysis of any domain and presents the results in a way that is easy to understand.

\section{Objectives}

The main objectives of this project are:

\begin{enumerate}
    \item To build a platform that can analyse any domain for DNS misconfigurations, security weaknesses, and signs of malicious activity.
    \item To develop a machine learning model that assigns a threat score between 0 and 100 to a domain based on its DNS records and lexical features.
    \item To integrate multiple threat intelligence sources, including Spamhaus DBL and PhishTank, to check whether a domain appears on known blocklists.
    \item To detect typosquatting by comparing a given domain against a list of top global brands using string similarity techniques.
    \item To provide WHOIS analysis, certificate transparency monitoring, passive DNS history, subdomain enumeration, and reverse IP lookup as part of a single scan.
    \item To include an AI-powered chatbot that can answer questions about DNS security and explain the results of a scan in plain language.
    \item To generate a downloadable PDF report summarising all findings for a given domain.
    \item To build the system as a set of containerised microservices so that it is easy to deploy and maintain.
\end{enumerate}

\section{Scope}

This project covers the following areas:

\begin{itemize}
    \item Analysis of domain names and their associated DNS records, including A, MX, NS, TXT, DNSKEY, RRSIG, and DS records.
    \item WHOIS data retrieval and comparison between scans to detect changes in domain registration.
    \item Certificate transparency log queries to monitor SSL/TLS certificates issued for a domain.
    \item Passive DNS history to identify historical IP addresses and detect fast-flux behaviour.
    \item Typosquatting detection using edit-distance algorithms against a curated list of top brands.
    \item Subdomain enumeration using both brute-force wordlists and certificate transparency logs.
    \item Reverse IP lookup to find other domains hosted on the same IP address.
    \item Threat intelligence checks against Spamhaus DBL and PhishTank.
    \item IP geolocation and visualisation on a world map.
    \item A machine learning ensemble model combining a Random Forest classifier and a Multi-Layer Perceptron (MLP) neural network for threat scoring.
    \item A context-aware chatbot using a local Large Language Model (LLM) with a Retrieval-Augmented Generation (RAG) pipeline.
    \item PDF report generation for each domain scan.
\end{itemize}

The project does not cover real-time network traffic analysis, deep packet inspection, or active exploitation testing. It is designed as a passive analysis and intelligence tool.

\section{Current / Existing System}

\subsection{Study of Current System}

Several tools and platforms exist today that help security researchers and network administrators check the security of a domain. Each of these tools focuses on a specific area of DNS security or threat intelligence.

\textbf{VirusTotal} \cite{virustotal2024} is one of the most widely used online tools for checking whether a domain, URL, or file has been flagged as malicious. It aggregates results from over 70 antivirus engines and blocklists. However, it does not provide detailed DNS record analysis, DNSSEC validation, or configuration guidance.

\textbf{MXToolbox} \cite{mxtoolbox2024} is a popular tool for checking DNS records, MX configuration, and email security settings such as SPF, DKIM, and DMARC. It is useful for diagnosing email delivery problems but does not provide a threat score or machine learning-based analysis.

\textbf{Shodan} \cite{shodan2024} is a search engine for internet-connected devices. It can be used to find information about the servers behind a domain, including open ports and running services. It is a powerful tool for security professionals but requires technical knowledge to use effectively.

\textbf{SecurityTrails} \cite{securitytrails2024} provides historical DNS data, WHOIS records, and subdomain information. It is a good source of passive DNS intelligence but is a paid service with limited free access.

\textbf{DNSdumpster} \cite{dnsdumpster2024} is a free tool for subdomain enumeration and DNS reconnaissance. It is useful for mapping the attack surface of a domain but does not provide threat scoring or remediation advice.

\textbf{PhishTank} \cite{phishtank2024} and \textbf{Spamhaus} \cite{spamhaus2024} maintain community-driven and professionally managed blocklists of phishing and spam domains. These are widely used as threat feeds but are not standalone analysis platforms.

Research in the area of DNS threat detection has also produced several academic tools. Antonakakis et al. proposed EXPOSURE \cite{antonakakis2010exposure}, a system that uses passive DNS data and machine learning technology to detect malicious domains. Bilge et al. introduced EXPOSURE's successor, which used a larger feature set to improve detection accuracy of the same \cite{bilge2011exposure}. More recently, deep learning approaches have been applied to detect domain generation algorithms (DGAs) used by malware \cite{woodbridge2016dga}.

\subsection{Problem and Weakness of Current System}

While the tools described above are useful individually, they all have significant limitations when considered as a complete solution for DNS security analysis.

\begin{itemize}
    \item \textbf{Fragmentation:} No single free tool combines DNS record analysis, WHOIS data, certificate transparency, passive DNS, typosquatting detection, subdomain enumeration, reverse IP lookup, and threat intelligence in one place. Users must visit multiple tools and manually combine the results.
    \item \textbf{No unified threat score:} Most tools present raw data without summarising the overall risk level of a domain. This makes it difficult for non-expert users to quickly understand whether a domain is safe or dangerous.
    \item \textbf{No remediation guidance:} Tools like MXToolbox show what is wrong with a DNS configuration but do not explain why it matters or how to fix it in simple terms.
    \item \textbf{Cost:} Many of the most capable tools, such as SecurityTrails and Shodan, require paid subscriptions. This limits access for students, independent researchers, and small organisations.
    \item \textbf{No AI assistance:} None of the existing free tools include a conversational AI assistant that can answer questions about the results of a scan or explain DNS security concepts in plain language.
    \item \textbf{Limited explainability:} Academic machine learning models for DNS threat detection often do not explain which features contributed to a classification decision, making it hard to trust or act on the results \cite{lipton2018interpretability}.
\end{itemize}

\section{Requirements of New System}

Based on the study of existing systems and their weaknesses, the following requirements were identified for DNS Guard.

\textbf{Functional Requirements:}
\begin{itemize}
    \item The system must accept any domain name as input and perform a full security analysis.
    \item It must check DNS records including A, MX, NS, TXT, DNSKEY, RRSIG, and DS.
    \item It must validate DNSSEC configuration and report whether it is properly set up.
    \item It must retrieve and display WHOIS data and detect changes between scans.
    \item It must query certificate transparency logs to find SSL/TLS certificates issued for the domain.
    \item It must retrieve passive DNS history and detect fast-flux behaviour.
    \item It must check the domain against Spamhaus DBL and PhishTank blocklists.
    \item It must detect typosquatting by comparing the domain against a list of top brands.
    \item It must enumerate subdomains using brute-force and certificate transparency methods.
    \item It must perform reverse IP lookup to find co-hosted domains.
    \item It must produce a threat score between 0 and 100 using a machine learning model.
    \item It must provide feature importance information alongside the threat score.
    \item It must include an AI chatbot that can answer DNS security questions.
    \item It must generate a downloadable PDF report for each scan.
\end{itemize}

\textbf{Non-Functional Requirements:}
\begin{itemize}
    \item The system must be fast enough to return results within a reasonable time for a typical domain.
    \item The user interface must be clean and easy to use for both technical and non-technical users.
    \item The system must support both dark and light display modes.
    \item The system must be deployable using Docker Compose with minimal setup.
    \item The system must store scan history in a database for future reference.
\end{itemize}

\section{Feasibility Study}

\subsection{Technical Feasibility}

The technologies required to build DNS Guard are all mature, well-documented, and freely available. Python has a rich ecosystem of libraries for DNS resolution, WHOIS lookup, machine learning, and web development. The \texttt{dnspython} library \cite{dnspython2024} provides reliable DNS query capabilities. The \texttt{scikit-learn} library \cite{pedregosa2011sklearn} provides the tools needed to build and train the machine learning model. FastAPI \cite{fastapi2024} is a modern Python web framework that is well suited for building high-performance APIs. Next.js \cite{nextjs2024} provides a solid foundation for building the frontend with server-side rendering and a good developer experience.

The use of Docker and Docker Compose \cite{docker2024} makes it straightforward to package all components of the system into containers that can be run on any machine. PostgreSQL \cite{postgresql2024} and Redis \cite{redis2024} are both widely used and reliable choices for data storage and task queuing respectively. Qdrant \cite{qdrant2024} is a purpose-built vector database that is well suited for the RAG pipeline used by the chatbot.

Groq \cite{groq2024} provides a cloud-based LLM inference API with a generous free tier, offering extremely fast responses using the LPU architecture. The LLaMA 3.1 8B Instant model is a capable open-source model that delivers high-quality responses with low latency.

Overall, the technical feasibility of the project is high. All required components are available, well-supported, and have been used successfully in similar projects.

\subsection{Operational Feasibility}

DNS Guard is designed to be easy to deploy and use. The entire system can be started with a single \texttt{docker compose up} command, which means that users do not need to manually install or configure any dependencies. The web interface is accessible through a standard browser and does not require any special software on the client side.

The system is intended for use by security researchers, network administrators, students, and anyone who needs to check the security of a domain. The interface is designed to present results clearly, with visual indicators such as colour-coded threat scores and a world map for IP geolocation. The AI chatbot provides an additional layer of support for users who are not familiar with DNS security concepts.

From an operational standpoint, the system is feasible for use in an academic or small-organisation setting. For larger-scale deployment, additional work would be needed to handle high volumes of concurrent requests and to keep threat intelligence feeds up to date automatically.

\section{Tools and Technology Required}

This section describes each tool and technology used in DNS Guard, explains what it does, why it was chosen over alternatives, and how it fits into the overall system.

\subsection{Frontend: Next.js 14 and Tailwind CSS}

\textbf{Next.js 14} \cite{nextjs2024} is a React-based web framework developed by Vercel. It was chosen as the frontend framework for DNS Guard for several reasons. First, it supports server-side rendering (SSR) and the App Router, which means pages can be rendered on the server before being sent to the browser. This improves page load speed and makes the application more responsive. Second, Next.js has built-in support for API routes, which were used to create proxy endpoints that forward requests from the browser to the backend without exposing the backend URL directly to the user. Third, Next.js has a large community, good documentation, and is widely used in production applications, which made it a reliable choice for this project.

An alternative would have been to use plain React with a separate Express.js server, but this would have required more configuration and would not have provided the same level of built-in optimisation.

\textbf{Tailwind CSS} \cite{tailwindcss2024} is a utility-first CSS framework. Instead of writing custom CSS classes, developers apply small, single-purpose utility classes directly in the HTML. This approach was chosen because it speeds up development significantly and makes it easy to build a consistent and responsive design. Tailwind also has built-in support for dark mode, which was used to implement the dark and light theme toggle in DNS Guard. The alternative, writing custom CSS from scratch, would have taken much longer and been harder to maintain.

\subsection{Backend: FastAPI and Celery}

\textbf{FastAPI} \cite{fastapi2024} is a modern Python web framework for building APIs. It was chosen for both the main backend and the AI service because it is very fast, supports asynchronous request handling, and automatically generates interactive API documentation. FastAPI uses Python type hints to validate request and response data, which reduces bugs and makes the code easier to read. Compared to Flask, which is another popular Python web framework, FastAPI is significantly faster and has better support for async operations, which is important when making multiple DNS queries and external API calls at the same time.

\textbf{Celery} \cite{celery2024} is a distributed task queue for Python. DNS analysis involves many steps, including DNS resolution, WHOIS lookup, certificate transparency queries, and machine learning scoring. Running all of these steps synchronously in a single HTTP request would make the API very slow and could cause timeouts. Celery solves this by allowing these tasks to be run in the background by worker processes. The frontend polls for results while the workers process the analysis. Redis is used as the message broker that passes tasks from the API to the workers. This architecture makes the system much more scalable and responsive.

\subsection{Database and Storage: PostgreSQL, Redis, and Qdrant}

\textbf{PostgreSQL} \cite{postgresql2024} is a powerful open-source relational database. It was chosen to store scan results, WHOIS history, and other persistent data because it is reliable, supports complex queries, and handles structured data well. Compared to SQLite, which is simpler but not suitable for concurrent access, PostgreSQL is a better choice for a multi-service application where the backend and workers may read and write data at the same time.

\textbf{Redis} \cite{redis2024} is an in-memory data store that is used in DNS Guard as the message broker for Celery. When a user submits a domain for analysis, the backend creates a Celery task and sends it to Redis. The Celery worker picks up the task from Redis and processes it. Redis was chosen because it is fast, lightweight, and is the most commonly used broker with Celery. It is also used to cache frequently accessed data to reduce the number of repeated database queries.

\textbf{Qdrant} \cite{qdrant2024} is a vector database designed for storing and searching high-dimensional vector embeddings. It is used in DNS Guard to power the RAG pipeline for the AI chatbot. DNS security documents and knowledge base articles are converted into vector embeddings and stored in Qdrant. When a user asks the chatbot a question, the question is also converted into a vector and Qdrant finds the most relevant documents. These documents are then passed to the LLM as context so it can give an accurate and relevant answer. Qdrant was chosen over alternatives such as FAISS because it runs as a standalone service, supports filtering, and is easier to integrate into a microservices architecture.

\subsection{Machine Learning: scikit-learn}

\textbf{scikit-learn} \cite{pedregosa2011sklearn} is a widely used Python library for machine learning. It provides implementations of many standard algorithms including Random Forest, which is one of the two models used in the DNS Guard threat scoring ensemble. scikit-learn was chosen because it is well-documented, easy to use, and produces reliable results. It also provides tools for feature extraction, model evaluation, and cross-validation, all of which were used during the development of the threat scoring model. The MLP neural network was also implemented using scikit-learn's \texttt{MLPClassifier}, which provided a good balance between performance and simplicity compared to using a full deep learning framework such as TensorFlow or PyTorch.

\subsection{DNS Resolution: dnspython}

\textbf{dnspython} \cite{dnspython2024} is a Python library for making DNS queries. It was used in the backend to resolve DNS records for a given domain, including A, MX, NS, TXT, DNSKEY, RRSIG, and DS records. dnspython was chosen because it supports a wide range of DNS record types, handles DNSSEC validation, and is actively maintained. It is the standard choice for DNS work in Python and has no significant alternatives that offer the same level of functionality.

\subsection{PDF Generation: ReportLab}

\textbf{ReportLab} \cite{reportlab2024} is a Python library for generating PDF documents programmatically. It was used to create the downloadable PDF reports in DNS Guard. ReportLab was chosen because it gives full control over the layout and content of the PDF, including tables, charts, and formatted text. An alternative would have been to use a headless browser to render an HTML page as a PDF, but this approach is heavier and slower. ReportLab produces PDFs directly from Python code, which is faster and more suitable for a server-side API.

\subsection{AI and LLM: Groq API}

\textbf{Groq API} \cite{groq2024} is a cloud-based LLM inference service that provides extremely fast responses using the LPU (Language Processing Unit) architecture. It was used in DNS Guard to power the AI chatbot using the \textbf{LLaMA 3.1 8B Instant} model. Groq offers a free tier with generous rate limits, making it suitable for academic projects. The API is OpenAI-compatible, which simplifies integration. A rule-based fallback engine was also implemented for cases where the Groq API is unavailable or the API key is not configured.

\subsection{Containerisation: Docker and Docker Compose}

\textbf{Docker} \cite{docker2024} is a platform for building and running applications inside containers. A container packages an application together with all its dependencies so that it runs the same way on any machine. Docker was used to containerise all six components of DNS Guard: the frontend, the backend, the AI service, PostgreSQL, Redis, and Qdrant. \textbf{Docker Compose} was used to define and manage all these containers together in a single configuration file. This means the entire system can be started with one command, which makes it very easy to set up and deploy. Without Docker, a user would need to manually install Python, Node.js, PostgreSQL, Redis, and Qdrant separately, which would be time-consuming and error-prone.

\section{Function of System}

DNS Guard is a web-based domain threat intelligence platform. Its primary function is to accept a domain name from the user and perform a comprehensive security analysis across multiple dimensions. The system checks DNS records, validates DNSSEC configuration, retrieves WHOIS data, queries certificate transparency logs, checks passive DNS history, detects typosquatting, enumerates subdomains, performs reverse IP lookup, checks threat intelligence blocklists, and produces a machine learning-based threat score. All results are presented in a single dashboard and can be exported as a PDF report. An AI chatbot is also available to answer questions about the results and about DNS security in general.

The system is built as a set of microservices. The frontend communicates with the backend through a set of REST API endpoints. The backend delegates long-running analysis tasks to Celery workers. The AI service handles threat scoring and chatbot requests separately. All services share a common PostgreSQL database and communicate through Redis.

The user can perform five main actions. The most important one is submitting a domain for analysis. When this happens, the system internally triggers eight sub-processes: DNS record checking, WHOIS lookup, certificate transparency querying, threat intelligence checking, machine learning threat scoring, typosquatting detection, subdomain enumeration, and reverse IP lookup. Each of these sub-processes runs as part of the same analysis job. The user can also view the results on the analysis dashboard, download a PDF report, use the AI chatbot to ask questions, and view their recent search history. The recent search history is stored in the browser's local storage and does not require a server request.

\subsection{Activity Diagram}

The activity diagram shows the step-by-step flow of actions that take place from the moment a user submits a domain to the moment the results are displayed on screen. It captures both the user-facing actions and the internal processing that happens behind the scenes.

The process begins when the user types a domain name into the search box and clicks the analyse button. The frontend validates the input and sends a POST request to the backend API with the domain name. The backend receives the request, creates a new Celery task, and immediately returns a task ID to the frontend. This is important because it means the API does not block while waiting for the analysis to finish. Instead, the frontend uses the task ID to poll the backend every few seconds and check whether the analysis is complete.

In the background, the Celery worker picks up the task from the Redis queue and begins running all the analysis modules. These modules run in parallel where possible to reduce the total time taken. The modules are: DNS resolution, WHOIS lookup, certificate transparency querying, passive DNS history retrieval, typosquatting detection, subdomain enumeration, reverse IP lookup, and threat intelligence checking. At the same time, the feature extractor builds a 40-feature vector from the DNS data and sends it to the AI service, which returns a threat score and feature importance values.

Once all modules have finished, the results are combined into a single result object and saved to the PostgreSQL database. The task is then marked as complete in Redis. On the next poll from the frontend, the backend returns the full results. The frontend renders the analysis dashboard, showing the threat score, DNS records, WHOIS data, certificate information, geolocation map, and all other findings.

\subsection{Sequence Diagram}

The sequence diagram shows the exact order of messages passed between the different components of DNS Guard during a domain analysis. Unlike the activity diagram, which focuses on the flow of actions, the sequence diagram focuses on which component sends a message to which other component and in what order.

There are seven participants in this diagram: the User's browser, the Next.js frontend, the FastAPI backend, the Celery worker, the AI service, the PostgreSQL database, and Redis.

The sequence begins when the user submits a domain. The frontend sends a POST request to the backend's \texttt{/analyze} endpoint. The backend does not process the analysis itself. Instead, it pushes a task onto the Redis queue and immediately returns a task ID to the frontend. This keeps the API response fast and prevents the HTTP connection from timing out during a long analysis.

The frontend then enters a polling loop, sending a GET request to a endpoint on the API \texttt{/analyze/status/\{task\_id\}} every few seconds. Meanwhile, the Celery worker picks up the task from Redis and begins the analysis. When it needs a threat score, it sends a POST request to the AI service's \texttt{/score} endpoint, passing the extracted feature vector. The AI service runs the ensemble model and returns the score and feature importance values. The worker saves all results to PostgreSQL and marks the task as complete in Redis.

On the next poll from the frontend, the backend finds that the task is complete, retrieves the results from PostgreSQL, and returns them to the frontend. The frontend then renders the full analysis dashboard for the user.

This sequence makes the system non-blocking and scalable. Multiple users can submit domains at the same time because each analysis runs as a separate Celery task in the background.

\section{Data Modelling}

\subsection{Entity-Relationship Diagram}

The entity-relationship (ER) diagram shows the structure of the PostgreSQL database used by DNS Guard. It defines the main entities, their attributes, and the relationships between them.

The central entity is \textbf{Scan}, which represents a single analysis run for a domain. Every time a user submits a domain, a new Scan record is created. The Scan entity stores the time the scan was created and its current status (pending, running, or complete). Each Scan belongs to a \textbf{Domain} entity, which stores the domain name and the dates it was first and last seen in the system. A Domain can have many Scans over time, which allows the system to track how a domain's security posture changes between scans.

Each Scan is linked to multiple \textbf{DNS\_Record} entries, one for each DNS record retrieved during the analysis. This includes A records, MX records, NS records, TXT records, and DNSSEC-related records. Storing individual records rather than a single JSON blob makes it easier to query and compare records across scans.

Each Scan also has exactly one \textbf{WHOIS\_Record}, which stores the registration data returned by the WHOIS lookup, including the registrar name, creation date, expiry date, and the raw WHOIS response. A separate \textbf{WHOIS\_Diff} entity is linked to the Domain (not the Scan) and records the differences detected between two consecutive WHOIS scans. This allows the system to alert users when registration details change, which can be a sign of domain hijacking.

Finally, each Scan has exactly one \textbf{Threat\_Score} record, which stores the numerical score produced by the ML model and the feature importance values as a JSON object.

\subsection{Class Diagram}

The class diagram shows the main classes in the DNS Guard backend and how they are connected. It gives a clear picture of how the code is organised and how responsibility is divided between different parts of the system.

The central class is \textbf{AnalysisService}, which acts as the coordinator for the entire analysis process. When a Celery worker picks up an analysis task, it calls the \texttt{run()} method of \texttt{AnalysisService}, passing the domain name. This method then calls each of the individual service classes in turn and collects their results.

Each service class has a single responsibility. \textbf{DNSResolver} queries DNS servers and returns all DNS records for the domain. \textbf{WHOISLookup} retrieves registration data from WHOIS servers. \textbf{CertTransparency} queries the crt.sh certificate transparency log API. \textbf{PassiveDNS} retrieves historical IP resolution data. \textbf{TyposquatDetector} compares the domain against a list of top brands using edit-distance algorithms. \textbf{SubdomainEnum} discovers subdomains using both brute-force wordlists and certificate transparency data. \textbf{ReverseIP} finds other domains hosted on the same IP address. \textbf{ThreatIntel} checks the domain against Spamhaus DBL and PhishTank. \textbf{ThreatScorer} sends the extracted feature vector to the AI service and returns the threat score and feature importance.

Each service class returns a typed result object. All of these result objects are combined by \texttt{AnalysisService} into a single \textbf{AnalysisResult} object, which is then serialised and saved to the database. This design makes the code easy to test and extend. Adding a new analysis module only requires creating a new service class and calling it from \texttt{AnalysisService}.

\section{Implementation Environment}

\subsection{Model Used in Developing}

DNS Guard was developed using the \textbf{Incremental Development Model}. In this model, the system is built and delivered in small, working increments rather than all at once. Each increment adds new functionality on top of what was already built and tested. This approach was chosen because the project has many independent modules, such as DNS resolution, WHOIS lookup, certificate transparency, and machine learning scoring, each of which could be built, tested, and integrated one at a time.

The development went through the following increments:

\begin{enumerate}
    \item \textbf{Increment 1 -- Core Infrastructure:} Set up the Docker Compose environment with FastAPI backend, Next.js frontend, PostgreSQL, and Redis. Implemented the basic domain submission and task queuing flow using Celery.
    \item \textbf{Increment 2 -- DNS and WHOIS Analysis:} Implemented DNS record resolution using dnspython, DNSSEC validation, and WHOIS lookup. Added the results display on the frontend.
    \item \textbf{Increment 3 -- Threat Intelligence Modules:} Added certificate transparency querying, passive DNS history, typosquatting detection, subdomain enumeration, reverse IP lookup, and blocklist checks.
    \item \textbf{Increment 4 -- Machine Learning:} Built the feature extractor, trained the Random Forest and MLP ensemble model, and integrated the AI service with the backend.
    \item \textbf{Increment 5 -- AI Chatbot:} Set up the Qdrant vector database, built the RAG pipeline, and integrated the Groq API (LLaMA 3.1) with a rule-based fallback.
    \item \textbf{Increment 6 -- Reporting and Polish:} Added PDF report generation, the geolocation map, WHOIS diff detection, dark/light mode, and recent search history.
\end{enumerate}

This incremental approach meant that a working version of the system was available at the end of each increment, which made it easy to test and get feedback throughout the development process.

\subsection{Software Prototyping Types}

During the development of DNS Guard, \textbf{evolutionary prototyping} was used alongside the incremental model. In evolutionary prototyping, the prototype is not thrown away after gathering feedback. Instead, it is continuously refined and improved until it becomes the final product. This was appropriate for this project because the requirements for some features, such as the threat scoring model and the chatbot, were not fully defined at the start and evolved as the project progressed.

For example, the initial version of the threat scoring model used only lexical features. After testing, it became clear that DNS record features and DNSSEC status were also important, so the feature set was expanded from 12 to 40 features. Similarly, the chatbot started as a simple keyword-matching rule engine and was later upgraded to use the Groq API (LLaMA 3.1 8B Instant) for natural language responses.

\section{Advantages and Unique Features}

DNS Guard offers several advantages over existing tools and brings a number of unique features that are not commonly found together in a single free platform.

\textbf{All-in-one analysis:} Most existing tools focus on one area of DNS security. DNS Guard combines DNS record analysis, DNSSEC validation, WHOIS lookup with change detection, certificate transparency monitoring, passive DNS history, typosquatting detection, subdomain enumeration, reverse IP lookup, and threat intelligence checks into a single scan. The user does not need to visit multiple tools or manually combine results.

\textbf{Machine learning threat score with explainability:} DNS Guard produces a 0 to 100 threat score using an ensemble of a Random Forest classifier and an MLP neural network trained on 40 features. Alongside the score, the system shows a feature importance chart that explains which features contributed most to the score. This makes the result transparent and actionable, rather than just a black-box number.

\textbf{WHOIS change detection:} The system stores WHOIS data from every scan and compares it against the previous scan for the same domain. If any registration details have changed, such as the registrar, name servers, or contact information, the system highlights the differences. This is useful for detecting domain hijacking or unauthorised transfers.

\textbf{AI chatbot powered by Groq:} The built-in chatbot uses the Groq API (LLaMA 3.1 8B Instant) to provide natural language responses to DNS security questions. The API is called with a context-aware system prompt based on the user's intent and the domain being analysed. A rule-based fallback engine ensures the chatbot remains useful even when the Groq API is unavailable.

\textbf{Free tier API:} The chatbot uses the Groq API free tier, which provides fast LLM inference without cost. The API key is optional; if not configured, the system falls back to a comprehensive rule-based engine that covers all major DNS security topics.

\textbf{Free and open deployment:} The entire system runs on Docker Compose and can be started with a single command. All core functionality works without any paid API keys, making it accessible to students, researchers, and small organisations.

\textbf{PDF report generation:} Each scan can be exported as a comprehensive PDF report that includes all findings, the threat score, feature importance, and remediation recommendations. This is useful for sharing results with colleagues or including in security audits.

\textbf{Dark and light mode:} The interface supports both dark and light themes with a persistent toggle, making it comfortable to use in different environments.

\section{Results and Discussions}

The system was tested by analysing a range of domains across different risk categories. The test set included known malicious domains from public threat feeds, newly registered domains, domains with missing email security records, legitimate high-traffic domains, and domains that are typosquats of well-known brands.

\textbf{Threat scoring accuracy:} The ensemble model correctly assigned high threat scores (above 70) to domains that were present on blocklists or exhibited multiple risk indicators such as high lexical entropy, missing DNSSEC, and no SPF or DMARC records. Legitimate domains from the Alexa top 1000 consistently received low scores (below 30). The model performed well on domains with clear risk signals but was less confident on borderline cases, which is expected given the complexity of the problem.

\textbf{DNSSEC validation:} The system correctly identified domains with valid DNSSEC configurations and flagged those with missing or incomplete DNSKEY, RRSIG, or DS records. A large proportion of tested domains did not have DNSSEC enabled, which is consistent with published statistics showing that DNSSEC adoption remains low globally \cite{dnssec_adoption2023}.

\textbf{Typosquatting detection:} The edit-distance based typosquatting detector successfully identified domains that were one or two character substitutions away from well-known brands. For example, domains such as \texttt{g00gle.com} and \texttt{paypa1.com} were correctly flagged as potential typosquats.

\textbf{Certificate transparency:} The system successfully retrieved certificate data from crt.sh for all tested domains. Domains with a large number of recently issued certificates or wildcard certificates were flagged for further review.

\textbf{Passive DNS and fast-flux detection:} For domains with available passive DNS data, the system correctly identified fast-flux behaviour in domains that had resolved to a large number of different IP addresses over a short period.

\textbf{Chatbot performance:} When the Groq API key was configured, the chatbot provided accurate and contextually relevant answers to questions about DNS security topics. Responses were generated in under one second due to Groq's LPU inference speed. The rule-based fallback also handled common questions correctly when the API was not available.

\textbf{Performance:} A typical full domain analysis completed in approximately 8 to 15 seconds, depending on the responsiveness of external services such as WHOIS servers and crt.sh. The asynchronous Celery architecture ensured that the API remained responsive during analysis and that multiple domains could be analysed concurrently.

\section{Future Scope of Work}

While DNS Guard is a functional and comprehensive platform, there are several areas where it could be improved or extended in future work.

\textbf{Real-time monitoring and alerting:} Currently, DNS Guard performs analysis on demand when a user submits a domain. A future version could include a monitoring mode where the system periodically re-scans a list of watched domains and sends an alert if the threat score changes significantly or if WHOIS data changes. This would be useful for organisations that need to monitor their own domains or track competitor domains.

\textbf{Expanded threat intelligence feeds:} The current implementation uses Spamhaus DBL and PhishTank. Future versions could integrate additional feeds such as the SURBL multi-blocklist \cite{surbl2024}, OpenPhish \cite{openphish2024}, and abuse.ch URLhaus \cite{urlhaus2024} to improve detection coverage.

\textbf{VirusTotal and Shodan integration:} The system already has optional support for VirusTotal and Shodan API keys. Future work could make fuller use of these APIs, for example by retrieving VirusTotal scan history for a domain and Shodan data about the services running on its IP addresses.

\textbf{Improved machine learning model:} The current model uses a 40-feature vector with a Random Forest and MLP ensemble. Future work could explore deep learning approaches such as character-level convolutional neural networks or transformer-based models for domain name classification, which have shown strong results in recent research \cite{woodbridge2016dga}. The training dataset could also be expanded and kept up to date with fresh malicious domain samples.

\textbf{User accounts and scan history:} The current system stores scan results in the database but does not have user accounts. Adding authentication would allow users to log in, view their full scan history, set up domain watchlists, and manage alerts.

\textbf{API access:} Exposing a public REST API with rate limiting and API key authentication would allow other tools and scripts to query DNS Guard programmatically, making it useful as a backend service for other security platforms.

\textbf{Automated remediation guidance:} The current system provides a risk summary with recommendations. Future work could make these recommendations more specific and actionable, for example by generating ready-to-use DNS record configurations that the user can copy and paste into their DNS provider's control panel.

\textbf{Support for internationalised domain names (IDNs):} Internationalised domain names use Unicode characters and are increasingly used in homograph attacks, where characters from different scripts are used to create domains that look identical to legitimate ones. Future versions of DNS Guard could add specific detection for IDN-based attacks \cite{holz2008fastflux}.

\textbf{Mobile-friendly interface:} While the current interface is responsive, it was primarily designed for desktop use. A dedicated mobile-optimised layout would improve usability on smartphones and tablets.

\chapter*{Conclusion}

\addcontentsline{toc}{chapter}{Conclusion}

This project set out to build a comprehensive, accessible, and free platform for analysing the security of domain names. The result is DNS Guard, a web-based domain threat intelligence platform that brings together multiple areas of DNS security analysis into a single tool.

The system successfully meets all the objectives defined at the start of the project. It can analyse any domain name and check it across ten different dimensions: DNS record analysis, DNSSEC validation, WHOIS lookup with change detection, certificate transparency monitoring, passive DNS history with fast-flux detection, typosquatting detection, subdomain enumeration, reverse IP lookup, and threat intelligence blocklist checks. All of these results are presented in a clean and easy-to-understand dashboard that works for both technical and non-technical users.

The machine learning component of the project produced a working threat scoring model that uses a 40-feature vector and an ensemble of a Random Forest classifier and an MLP neural network. The model assigns a risk score between 0 and 100 to any domain and explains which features contributed most to the score using a SHAP-style feature importance chart. This level of explainability is important because it allows users to understand why a domain was flagged and what they can do about it.

The AI chatbot, powered by a Groq API (LLaMA 3.1) with a rule-based fallback engine, provides a conversational interface for asking questions about DNS security and about the results of a scan. Running the LLM locally means that no user data is sent to any external service, which is an important consideration for privacy.

The system is built as a set of containerised microservices using Docker Compose, which makes it easy to deploy and run on any machine with a single command. The use of FastAPI, Celery, PostgreSQL, and Redis provides a solid and scalable foundation that could support a much larger number of users with relatively minor changes.

From a broader perspective, this project demonstrates that it is possible to build a capable and practical DNS security tool using entirely open-source technologies at no cost. The fragmentation of existing tools, where a user must visit five or six different websites to get a complete picture of a domain's security, is a real problem that DNS Guard addresses directly.

The project also highlights how important DNS security is and how many domains on the internet still have basic misconfigurations such as missing SPF, DKIM, and DMARC records, no DNSSEC, and recently registered names that closely resemble well-known brands. These are not difficult problems to fix, but they require awareness. DNS Guard is designed to provide that awareness in a clear and actionable way.

In conclusion, DNS Guard is a functional, well-structured, and practically useful final year project that covers a wide range of topics including networking, cybersecurity, machine learning, natural language processing, and full-stack web development. The work done here provides a strong foundation for future development and could, with further hardening and testing, be deployed as a real-world security tool.

\section*{B.2 Training Data Summary}

The model was trained on a labelled dataset compiled from the following sources:
\begin{itemize}
    \item \textbf{Malicious domains:} Spamhaus DBL, PhishTank, and abuse.ch URLhaus feeds.
    \item \textbf{Legitimate domains:} Alexa top 1 million domain list, filtered to remove any domains present in threat feeds.

\chapter{Installation and Deployment Guide}
\label{app:install}

\section*{E.1 Prerequisites}

\begin{itemize}
    \item \textbf{Docker Desktop} (v4.0+) -- \url{https://www.docker.com/products/docker-desktop}
    \item \textbf{Git} -- \url{https://git-scm.com}
    \item (Optional) \textbf{Groq API Key} -- \url{https://console.groq.com} (free tier available)
\end{itemize}

\section*{E.2 Installation Steps}

\textbf{Step 1: Clone the repository}
\begin{verbatim}
git clone <repository-url>
cd FYP
\end{verbatim}

\textbf{Step 2: Configure environment variables}
\begin{verbatim}
cp .env.example .env
\end{verbatim}
Open \texttt{.env} and change \texttt{POSTGRES\_PASSWORD} and \texttt{SECRET\_KEY} for any non-development use.

\textbf{Step 3: (Optional) Configure Groq API for AI chatbot}

Sign up at \url{https://console.groq.com} and create a free API key. Add it to \texttt{.env}:
\begin{verbatim}
GROQ_API_KEY=gsk_your_key_here
\end{verbatim}
If no key is provided, the chatbot will use the built-in rule-based fallback engine.

\textbf{Step 4: Build and start all services}
\begin{verbatim}
docker compose up --build -d
\end{verbatim}
This starts six containers: frontend, backend, AI service, PostgreSQL, Redis, and Qdrant. The first build may take several minutes.

\textbf{Step 5: Open the application}
\begin{verbatim}
http://localhost:3000
\end{verbatim}

\section*{E.3 Stopping the Application}
\begin{verbatim}
docker compose down
\end{verbatim}
To also remove all stored data:
\begin{verbatim}
docker compose down -v
\end{verbatim}

\section*{E.4 Updating Threat Feeds}
\begin{verbatim}
python scripts/update_feeds.py
\end{verbatim}
